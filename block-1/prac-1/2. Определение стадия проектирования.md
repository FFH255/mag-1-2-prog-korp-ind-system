# Задание 2. Определение стадий проектирования

## 2. Определение стадий проектирования

### Стадия 1. Формирование требований к АС

| Параметр | Значение |
|---|---|
| **Назначение** | Обследование предприятия, выявление потребностей заказчика, формирование исходных требований к системе |
| **Входы** | Заявка на создание системы; нормативные документы предприятия; перечень ролей и процессов; описание предметной области |
| **Выходы** | Отчёт об обследовании; исходные требования (предварительные); технико-экономическое обоснование (ТЭО) |
| **Измеримый результат** | Подписанный заказчиком отчёт об обследовании; утверждённый перечень исходных требований объёмом не менее 20 пунктов |

**Разрывы ожиданий:**

| Разрыв | Описание |
|---|---|
| Полнота охвата | Заказчик может не осознавать, какие процессы подлежат автоматизации — необходимо явное согласование границ системы |
| Приоритеты | Разные роли (диспетчер, начальник участка) предъявляют конкурирующие требования — необходима фиксация приоритетов |

**Способы контроля:**

- Контрольная встреча с заказчиком по результатам обследования
- Подписание отчёта об обследовании

---

### Стадия 2. Разработка концепции АС

| Параметр | Значение |
|---|---|
| **Назначение** | Формирование и сравнение альтернативных концепций системы; выбор и обоснование архитектурного направления |
| **Входы** | Отчёт об обследовании; исходные требования; ТЭО |
| **Выходы** | Документ «Концепция АС»; описание альтернатив; рекомендуемый вариант реализации |
| **Измеримый результат** | Утверждённый документ «Концепция АС»; зафиксированное решение о выбранной архитектуре с обоснованием |

**Разрывы ожиданий:**

| Разрыв | Описание |
|---|---|
| Технологический выбор | Заказчик может настаивать на конкретной технологии без обоснования её пригодности |
| Масштаб | Концепция может расширяться за счёт пожеланий, не входящих в утверждённые границы |

**Способы контроля:**

- Проверка соответствия концепции зафиксированным границам из обследования
- Формальное согласование концепции с заказчиком, ИБ-специалистом и аналитиком

---

### Стадия 3. Техническое задание (ТЗ)

| Параметр | Значение |
|---|---|
| **Назначение** | Фиксация согласованных требований к системе в виде юридически значимого документа |
| **Входы** | Концепция АС; исходные требования; описание предметной области; перечень ролей |
| **Выходы** | Техническое задание (ТЗ) по ГОСТ 34.602-89 |
| **Измеримый результат** | Подписанное ТЗ; все разделы ТЗ заполнены; требования однозначны, проверяемы, не содержат противоречий |

**Разрывы ожиданий:**

| Разрыв | Описание |
|---|---|
| Детализация | Заказчик считает требования «очевидными» и не хочет их формализовывать |
| Изменения после подписания | Ожидание возможности вносить изменения без формального процесса согласования |

**Способы контроля:**

- Взаимная инспекция ТЗ аналитиком и архитектором
- Проверка ТЗ специалистом по ИБ на соответствие требованиям безопасности
- Подписание ТЗ всеми сторонами как условие перехода к следующей стадии

---

### Стадия 4. Эскизный проект

| Параметр | Значение |
|---|---|
| **Назначение** | Разработка предварительных проектных решений: структура данных, основные сценарии, укрупнённая архитектура |
| **Входы** | Утверждённое ТЗ |
| **Выходы** | Документ «Эскизный проект»: описание структуры, модели данных (концептуальная), Use Case-диаграммы |
| **Измеримый результат** | Согласованный эскизный проект; покрытие всех Use Cases из ТЗ не менее 100%; концептуальная модель данных утверждена |

**Разрывы ожиданий:**

| Разрыв | Описание |
|---|---|
| Детализация | Заказчик принимает эскизный проект за окончательный UI-макет |
| Полнота | Отдельные роли (специалист по ИБ) могут не участвовать в согласовании, оставляя требования безопасности непроверенными |

**Способы контроля:**

- Трассировка Use Cases к требованиям ТЗ
- Рецензия архитектора на концептуальную модель данных

---

### Стадия 5. Технический проект

| Параметр | Значение |
|---|---|
| **Назначение** | Разработка полных проектных решений: архитектура системы, логическая модель данных, спецификации интерфейсов, алгоритмы |
| **Входы** | Утверждённый эскизный проект; ТЗ |
| **Выходы** | Документ «Технический проект»: архитектурные схемы, логическая модель данных, API-спецификации, описание ролевой модели |
| **Измеримый результат** | Утверждённый технический проект; все требования ТЗ трассированы к проектным решениям; матрица трассируемости заполнена |

**Разрывы ожиданий:**

| Разрыв | Описание |
|---|---|
| Интеграции | Требования к интерфейсам с ERP могут быть неполными — необходимо явное согласование с владельцами внешних систем |
| ИБ | Ролевая модель и разграничение доступа могут быть не согласованы со специалистом по ИБ |

**Способы контроля:**

- Верификация технического проекта относительно ТЗ (матрица трассируемости)
- Обязательное согласование ролевой модели со специалистом по ИБ
- Инспекция архитектуры архитектором и ведущим разработчиком

---

### Стадия 6. Рабочая документация и разработка

| Параметр | Значение |
|---|---|
| **Назначение** | Создание рабочей документации и программного продукта на основе технического проекта |
| **Входы** | Утверждённый технический проект |
| **Выходы** | Исходный код; рабочая документация (руководство пользователя, руководство администратора, описание БД); программа и методика испытаний (ПМИ) |
| **Измеримый результат** | Все требования ТЗ реализованы; покрытие функциональности тестами не менее 80%; рабочая документация соответствует ГОСТ 34 |

**Разрывы ожиданий:**

| Разрыв | Описание |
|---|---|
| Качество кода | Разработка может опережать документирование — рабочая документация должна создаваться параллельно |
| Тестирование | Функциональное тестирование может быть сведено к минимуму при давлении сроков |

**Способы контроля:**

- Code review на каждый pull request
- Автоматизированное тестирование (unit + integration)
- Контроль соответствия рабочей документации финальному коду перед приёмкой

### Стадия 7. Ввод в действие

| Параметр | Значение |
|---|---|
| **Назначение** | Развёртывание системы, обучение персонала, проведение испытаний, приёмка заказчиком |
| **Входы** | Рабочая документация; программный продукт; ПМИ; акт готовности инфраструктуры |
| **Выходы** | Протоколы испытаний; акт приёмки в опытную эксплуатацию; акт приёмки в промышленную эксплуатацию |
| **Измеримый результат** | Все тесты ПМИ пройдены; акт приёмки подписан заказчиком; персонал прошёл обучение (подтверждено подписями) |

**Разрывы ожиданий:**

| Разрыв | Описание |
|---|---|
| Готовность инфраструктуры | Заказчик может не подготовить серверную среду к сроку развёртывания |
| Обучение | Пользователи могут не пройти обучение до начала опытной эксплуатации |

**Способы контроля:**

- Чек-лист готовности инфраструктуры (заполняется администратором ИС)
- Журнал обучения персонала
- Контрольный прогон всех сценариев ПМИ в присутствии заказчика

### Стадия 8. Сопровождение АС

| Параметр | Значение |
|---|---|
| **Назначение** | Поддержка эксплуатируемой системы: устранение дефектов, управление изменениями, обновления |
| **Входы** | Акт приёмки; рабочая документация; реестр дефектов и запросов на изменение |
| **Выходы** | Обновлённая документация; исправленные версии ПО; отчёты о сопровождении |
| **Измеримый результат** | Время реакции на критический дефект — не более 4 часов; все изменения фиксируются через процесс управления изменениями |

**Разрывы ожиданий:**

| Разрыв | Описание |
|---|---|
| Граница сопровождения | Заказчик воспринимает сопровождение как возможность добавлять новые функции без дополнительного ТЗ |

**Способы контроля:**

- Реестр запросов на изменение с разделением на дефекты и новые требования
- Ежеквартальный отчёт о состоянии системы
