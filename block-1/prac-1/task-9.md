## Дисциплина

**Программирование корпоративных индустриальных систем**

## Практическая работа №1

**Тема:** Идентификация стадий и этапов процесса проектирования ИС

## 9. Управление изменениями и трассируемость

---

### 9.1. Процесс управления изменениями требований

---

#### 9.1.1. Шаги процесса

```
[Инициатор]
    │
    │ Заполняет запрос на изменение (ЗИ)
    ▼
[1. Регистрация]
    Аналитик регистрирует ЗИ в реестре изменений
    Присваивает идентификатор: CR-YYYY-NNN
    Статус: «Зарегистрирован»
    │
    ▼
[2. Первичный анализ]
    Аналитик оценивает:
    — тип изменения (дефект / новое требование / уточнение)
    — затронутые артефакты
    — предварительный приоритет
    Срок: 2 рабочих дня
    │
    ├── [Тип: Дефект документации] → Аналитик исправляет, версия артефакта +0.1
    │
    └── [Тип: Изменение требования / Новое требование]
            │
            ▼
        [3. Оценка влияния]
            Аналитик + Архитектор оценивают:
            — затронутые требования
            — затронутые этапы и артефакты
            — оценку трудоёмкости изменения
            — риски внесения изменения
            Срок: 3 рабочих дня
            │
            ▼
        [4. Согласование]
            Решение принимает:
            — до подписания ТЗ: Аналитик + Заказчик
            — после подписания ТЗ: Аналитик + Заказчик + Архитектор + ИБ (при необходимости)
            │
            ├── [Отклонено] → Статус ЗИ: «Отклонён», причина фиксируется
            │
            └── [Одобрено]
                    │
                    ▼
                [5. Реализация]
                    Ответственный вносит изменения во все затронутые артефакты
                    Статус: «В работе»
                    │
                    ▼
                [6. Верификация]
                    Аналитик проверяет полноту внесённых изменений
                    Матрица трассируемости обновляется
                    │
                    ▼
                [7. Закрытие]
                    Статус ЗИ: «Закрыт»
                    Версии изменённых артефактов увеличиваются
```

---

#### 9.1.2. Реестр изменений (структура)

| Поле | Описание | Пример |
|---|---|---|
| ID | Уникальный идентификатор | CR-2025-001 |
| Дата регистрации | Дата поступления запроса | 15.10.2025 |
| Инициатор | Роль и имя инициатора | Заказчик |
| Описание | Суть предлагаемого изменения | Добавить поле «Категория оборудования» в заявку |
| Тип | Дефект / Новое требование / Уточнение | Новое требование |
| Затронутые артефакты | Список артефактов, требующих правки | A-07 ТЗ, A-10 ER-концептуальная, A-13 ER-логическая |
| Приоритет | Высокий / Средний / Низкий | Средний |
| Оценка трудоёмкости | Рабочие дни | 3 |
| Риск | Описание риска при внесении изменения | Нарушение 3НФ в логической модели |
| Решение | Одобрен / Отклонён / Отложен | Одобрен |
| Статус | Зарегистрирован / В работе / Закрыт / Отклонён | Закрыт |
| Дата закрытия | — | 20.10.2025 |
| Ответственный | Кто реализует | Аналитик |

---

#### 9.1.3. Приоритеты изменений

| Приоритет | Описание | Срок рассмотрения | Пример |
|---|---|---|---|
| **Высокий** | Изменение блокирует текущую работу или затрагивает безопасность | 1 рабочий день | Выявленная уязвимость в ролевой модели |
| **Средний** | Изменение значимо, но не блокирует работу | 3 рабочих дня | Добавление нового поля в сущность |
| **Низкий** | Косметическое или несущественное изменение | 5 рабочих дней | Переименование термина в документации |

---

#### 9.1.4. Правила версионирования при изменениях

- Любое изменение, прошедшее согласование, увеличивает версию артефакта на 0.1
- При изменении ≥ 30% содержания документа версия увеличивается на 1.0
- После каждого изменения артефакта обновляется лист согласования с датой и подписями
- Предыдущая версия хранится в архиве и не удаляется

### 9.2. Матрица трассируемости

---

#### 9.2.1. Назначение матрицы

Матрица трассируемости обеспечивает:
- проверку того, что каждое требование реализовано
- выявление «сирот» — артефактов без источника требования
- оценку влияния изменения требования на этапы и артефакты
- контроль полноты тестирования

---

#### 9.2.2. Структура матрицы трассируемости

| Требование ТЗ | Описание | UC (Эскизный проект) | Проектное решение (Тех.проект) | Артефакт (Разработка) | Тест-кейс (ПМИ) | Статус |
|---|---|---|---|---|---|---|
| REQ-F-001 | Создание заявки диспетчером | UC-01 | Компонент WorkorderService; таблица `work_orders` | A-17 (модуль создания заявок) | TC-001 | ✅ Реализовано |
| REQ-F-002 | Назначение исполнителя на заявку | UC-02 | WorkorderService.assignEngineer(); таблица `assignments` | A-17 (модуль назначения) | TC-002 | ✅ Реализовано |
| REQ-F-003 | Изменение статуса заявки | UC-03, UC-04 | WorkorderService.updateStatus(); таблица `work_order_statuses` | A-17 (модуль статусов) | TC-003, TC-004 | ✅ Реализовано |
| REQ-F-004 | Формирование отчёта о выполнении | UC-05 | ReportService; таблица `work_reports` | A-17 (модуль отчётов) | TC-005 | ✅ Реализовано |
| REQ-F-005 | Просмотр истории изменений заявки | UC-06 | AuditService; таблица `audit_log` | A-17 (модуль аудита) | TC-006 | ✅ Реализовано |
| REQ-F-006 | Эскалация просроченных заявок | UC-07 | EscalationService; CRON-задача | A-17 (модуль эскалации) | TC-007 | ✅ Реализовано |
| REQ-F-007 | Формирование аналитических отчётов | UC-08 | AnalyticsService; представления БД | A-17 (модуль аналитики) | TC-008 | ✅ Реализовано |
| REQ-NF-001 | Разграничение доступа по ролям | — | Матрица доступа (A-15); AuthService | A-17 (auth-модуль) | TC-009 | ✅ Реализовано |
| REQ-NF-002 | Аудит действий пользователей | — | AuditService; таблица `audit_log` | A-17 (модуль аудита) | TC-010 | ✅ Реализовано |
| REQ-NF-003 | Резервное копирование данных | — | Инфраструктурное решение (A-19) | A-22 (чек-лист инфраструктуры) | TC-011 | ✅ Реализовано |

*Примечание: таблица является шаблоном; конкретные идентификаторы UC, компонентов и тест-кейсов уточняются на соответствующих стадиях проекта.*

---

#### 9.2.3. Правила ведения матрицы трассируемости

| Правило | Описание |
|---|---|
| Полнота | Каждое требование ТЗ (функциональное и нефункциональное) должно присутствовать в матрице |
| Актуальность | Матрица обновляется при каждом изменении требований, проектных решений или тест-кейсов |
| Сирота | Требование без UC, проектного решения или тест-кейса — дефект, требующий устранения до перехода на следующую стадию |
| Ответственный | Аналитик несёт ответственность за актуальность матрицы |
| Проверка | Матрица проверяется на каждой контрольной точке (КТ-3 — КТ-7) |

### 9.3. Правила разрешения конфликтов

---

#### 9.3.1. Виды конфликтов

| Вид конфликта | Описание | Пример |
|---|---|---|
| **Конфликт требований** | Два требования ТЗ логически противоречат друг другу | REQ-F-001: «диспетчер создаёт заявку» vs REQ-F-010: «только начальник участка создаёт заявку» |
| **Конфликт интересов** | Разные роли предъявляют несовместимые требования | Диспетчер хочет упрощённую форму; ИБ требует обязательных полей |
| **Конфликт технического решения** | Архитектор предлагает решение, которое не устраивает разработчика по сложности реализации | Выбор ORM vs raw SQL |
| **Конфликт приоритетов** | Изменения конкурируют за ресурс разработчика | CR-2025-001 (высокий) и CR-2025-002 (высокий) требуют один и тот же модуль |

---

#### 9.3.2. Процесс разрешения конфликтов

```
[Конфликт выявлен]
    │
    ▼
[1. Фиксация]
    Аналитик фиксирует конфликт в реестре конфликтов
    Описание: требования A и B, суть противоречия
    │
    ▼
[2. Эскалация по уровням]
    Уровень 1: Аналитик пытается разрешить через уточнение требований
               Срок: 1 рабочий день
    │
    ├── [Разрешено] → Вносятся правки в артефакты, фиксируется решение
    │
    └── [Не разрешено]
            │
            ▼
        Уровень 2: Аналитик + Архитектор + Заказчик
                   Совещание по конфликту
                   Срок: 2 рабочих дня
            │
            ├── [Разрешено] → Вносятся правки, решение фиксируется с подписями
            │
            └── [Не разрешено]
                    │
                    ▼
                Уровень 3: Руководитель проекта / Спонсор
                           Административное решение
                           Срок: 3 рабочих дня
```

---

#### 9.3.3. Приоритет при разрешении конфликтов требований

| Приоритет | Тип требования |
|---|---|
| 1 (высший) | Требования безопасности (ИБ) |
| 2 | Требования законодательства и нормативных документов |
| 3 | Функциональные требования заказчика (A в RACI) |
| 4 | Нефункциональные требования (производительность, надёжность) |
| 5 (низший) | Требования удобства использования (UX) |

При конфликте выигрывает требование с более высоким приоритетом. Если приоритеты равны — решение принимается заказчиком с фиксацией обоснования.
