### 8.2. Чек-лист самопроверки (для каждой стадии)

Перед предъявлением результатов стадии исполнитель обязан проверить следующее:

#### Общий чек-лист (применим ко всем стадиям)

- [ ] Все запланированные артефакты созданы
- [ ] Артефакты соответствуют требованиям к формату и минимальному объёму (см. task-4.md)
- [ ] Имена файлов соответствуют правилам именования
- [ ] Версия артефакта актуальна (не черновик, если предполагается согласование)
- [ ] Все требуемые подписи получены
- [ ] Матрица трассируемости обновлена
- [ ] Реестр рисков проверен (новые риски зафиксированы)

#### Специфический чек-лист для стадии «Требования»

- [ ] Каждое требование начинается с «Система должна...» или «Пользователь должен...»
- [ ] Отсутствуют требования с неопределёнными словами: «быстро», «удобно», «при необходимости»
- [ ] Каждое требование верифицируемо без устных пояснений

#### Специфический чек-лист для стадии «Проектирование»

- [ ] Каждое требование ТЗ трассировано к проектному артефакту
- [ ] Ни одна сущность предметной области не потеряна в моделях
- [ ] Ролевая модель проверена специалистом по ИБ

#### Специфический чек-лист для стадии «Разработка»

- [ ] Отчёт покрытия тестами показывает ≥ 80%
- [ ] Все pull request прошли code review
- [ ] Документация соответствует реализованному коду

### 8.3. Типовые ошибки при сдаче стадий

| Тип ошибки | Описание | Стадия появления | Способ устранения |
|---|---|---|---|
| **Несогласованная терминология** | Один объект называется по-разному в ТЗ, эскизном и техническом проекте (например, «заявка» и «запрос на ремонт») | 3–5 | Ведение глоссария с первой стадии; проверка терминологии при согласовании каждого документа |
| **Отсутствие входных данных** | Документ ссылается на данные или решения, которые не зафиксированы ни в одном предыдущем артефакте | 3–7 | Проверка матрицы трассируемости перед сдачей |
| **Противоречия в требованиях** | Два требования ТЗ логически исключают друг друга | 3 | Инспекция ТЗ двумя независимыми ревьюерами (аналитик + архитектор) |
| **Неполное покрытие ролей** | Use Case-диаграммы, матрица RACI или документация не охватывают все 6 ролей | 4–6 | Проверка по чек-листу ролей перед сдачей |
| **Недокументированные проектные решения** | В техническом проекте принято решение, не отражённое в документе | 5 | Требование: каждое архитектурное решение фиксируется с обоснованием |
| **Отсутствие трассируемости** | Требование ТЗ не имеет соответствующего UC, проектного решения или тест-кейса | 4–6 | Обязательное ведение и проверка матрицы трассируемости |
| **Непроверяемые критерии приёмки** | Тест-кейс ПМИ содержит субъективный ожидаемый результат («должно работать хорошо») | 6 | Каждый ожидаемый результат описывается конкретным измеримым состоянием системы |
| **Неактуальная документация** | Руководство пользователя описывает функциональность, изменённую в процессе разработки | 6–7 | Обязательная синхронизация документации с кодом перед КТ-6 |
