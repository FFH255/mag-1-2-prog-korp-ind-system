# Задание 10. Реестр рисков

## Дисциплина

**Программирование корпоративных индустриальных систем**

## Практическая работа №1

**Тема:** Идентификация стадий и этапов процесса проектирования ИС

**Предметная область:** Система управления заявками на обслуживание и ремонт оборудования (ТОиР)

**Нормативная основа:** ГОСТ 34.601-90 + ГОСТ Р 59792-2021 (ISO/IEC 12207:2017)

## 10. Реестр рисков

---

### 10.1. Шкалы оценки

#### Вероятность (P)

| Значение | Уровень | Описание |
|---|---|---|
| 1 | Низкая | Риск маловероятен; редкое явление для проектов данного типа |
| 2 | Средняя | Риск вероятен; периодически наблюдается в аналогичных проектах |
| 3 | Высокая | Риск почти неизбежен при данных условиях |

#### Ущерб (I)

| Значение | Уровень | Описание |
|---|---|---|
| 1 | Незначительный | Незначительная задержка (≤ 3 дня) или исправимые дефекты |
| 2 | Существенный | Задержка стадии (4–15 дней) или переработка значимых артефактов |
| 3 | Критический | Срыв сроков (> 15 дней), потеря данных, провал приёмки |

#### Рейтинг риска (R = P × I)

| R | Уровень | Действие |
|---|---|---|
| 1–2 | Низкий | Мониторинг |
| 3–4 | Средний | Планирование меры реагирования |
| 6–9 | Высокий | Немедленные превентивные меры |

#### РИСК-01: Неполнота или противоречивость требований заказчика

| Параметр | Значение |
|---|---|
| **Категория** | Качество данных / требований |
| **Описание** | Представители заказчика предоставляют неполную или противоречивую информацию в ходе обследования, что приводит к неверным требованиям в ТЗ |
| **Вероятность (P)** | 3 — Высокая |
| **Ущерб (I)** | 3 — Критический |
| **Рейтинг (R)** | 9 — Высокий |
| **Стадия выявления** | Стадия 1 (Формирование требований) |
| **Индикаторы** | Противоречия между протоколами интервью разных ролей; требования, изменяющиеся от встречи к встрече |
| **Меры** | 1. Обязательное согласование перечня требований со всеми ролями до закрытия стадии 1. 2. Фиксация требований в письменном виде с подписью. 3. Проведение совместного совещания при выявлении противоречий |
| **Подтверждающий артефакт** | A-03 Перечень исходных требований (наличие подписей всех ролей) |

#### РИСК-03: Сложность интеграции с ERP-системой

| Параметр | Значение |
|---|---|
| **Категория** | Интеграции |
| **Описание** | ERP-система предприятия имеет нестандартный интерфейс или отсутствующую документацию API, что делает интеграцию более трудоёмкой или невозможной в рамках проекта |
| **Вероятность (P)** | 2 — Средняя |
| **Ущерб (I)** | 3 — Критический |
| **Рейтинг (R)** | 6 — Высокий |
| **Стадия выявления** | Стадия 2 (Концепция АС) |
| **Индикаторы** | Отсутствие документации API ERP; изменение версии ERP в ходе проекта; отказ ИТ-службы предоставить тестовый доступ |
| **Меры** | 1. На стадии концепции явно зафиксировать доступность API ERP как предусловие интеграции. 2. Предусмотреть вариант «без интеграции» с ручным экспортом данных. 3. Получить тестовый доступ к ERP до подписания ТЗ |
| **Подтверждающий артефакт** | A-06 Концепция АС (раздел «Интеграции»); A-07 ТЗ (раздел «Внешние системы») |

#### РИСК-04: Срыв сроков из-за недоступности ключевых стейкхолдеров

| Параметр | Значение |
|---|---|
| **Категория** | Сроки |
| **Описание** | Заказчик или ключевые роли (начальник участка, специалист по ИБ) не могут участвовать в согласованиях в установленные сроки (командировки, отпуска, загруженность) |
| **Вероятность (P)** | 3 — Высокая |
| **Ущерб (I)** | 2 — Существенный |
| **Рейтинг (R)** | 6 — Высокий |
| **Стадия выявления** | Стадия 1, повторяется на всех стадиях |
| **Индикаторы** | Согласование документа задержано более чем на 3 рабочих дня сверх нормы; участник не отвечает на запросы |
| **Меры** | 1. Согласовать с заказчиком до начала проекта: перечень ответственных лиц, максимальные сроки согласования (SLA согласования). 2. Фиксировать срок ответа в каждом запросе. 3. Использовать риск-ориентированный план с буфером |
| **Подтверждающий артефакт** | A-07 ТЗ (раздел «Порядок контроля и приёмки») |

|---|---|

| **Описание** | Заказчик инициирует существенные изменения требований после подписания ТЗ, что нарушает базовую линию проекта и требует переработки уже выполненных артефактов |

| **Ущерб (I)** | 3 — Критический |
| **Рейтинг (R)** | 9 — Высокий |
| **Стадия выявления** | Стадия 3 (при подписании ТЗ); реализуется на стадиях 4–6 |
| **Индикаторы** | Более 3 запросов на изменение за стадию; запросы на изменение затрагивают ≥ 30% требований ТЗ |
| **Меры** | 1. В ТЗ явно зафиксировать процесс управления изменениями (ссылка на task-9). 2. Все изменения после подписания ТЗ оформляются только через ЗИ. 3. Оценка влияния каждого ЗИ на сроки и стоимость предъявляется заказчику до одобрения |
| **Подтверждающий артефакт** | A-07 ТЗ (раздел «Управление изменениями»); Реестр изменений (CR-log) |
